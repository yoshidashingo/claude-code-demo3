# 開発標準ルール

## 開発手法: 仕様駆動開発（Spec-Driven Development）

コードより先に仕様を書き、仕様がすべての実装・テスト・レビューの基準となる。

### 開発フロー

1. 要件整理
2. 仕様書作成（`specs/` に配置）
3. 仕様レビュー・承認
4. テストケース作成（仕様の受け入れ基準から導出）
5. 実装
6. 検証

### ルール

1. **仕様先行**: 仕様が承認されるまで実装コードを書かない
2. **仕様管理**: 仕様ファイルは `specs/` に機能単位で配置する（例: `specs/001-ユーザー認証.md`）
3. **受け入れ基準**: EARS記法で記述し、曖昧さを排除する
   - When [条件], the system shall [動作]
   - While [状態], the system shall [動作]
4. **テスト導出**: テストケースは仕様書の受け入れ基準から導出する。仕様にないテストは書かない
5. **変更管理**: 仕様変更は仕様ファイルを更新し、変更履歴セクションに記録する
6. **PR紐付け**: PRには対応する仕様ファイルへの参照を含める
7. **YAGNI**: 仕様にない機能を追加しない

### 仕様書テンプレート

```markdown
# 機能名

## 概要
（何を、なぜ作るか）

## 受け入れ基準（EARS記法）
- When [条件], the system shall [動作]
- While [状態], the system shall [動作]

## 非機能要件
（パフォーマンス、セキュリティ等）

## 依存関係
（他の仕様・外部サービスとの関係）

## 変更履歴
| 日付 | 変更内容 | 承認者 |
|------|----------|--------|
```

### ディレクトリ構成

```
specs/          # 仕様書
tests/          # 仕様から導出したテスト
src/            # 実装コード
```
